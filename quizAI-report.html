01

What Your Project Actually Does

Your project is a full-stack adaptive quiz platform. The Next.js frontend handles auth via Firebase and calls your Express.js backend (port 5000) with a Firebase JWT token. The backend validates requests via Zod middleware, checks MongoDB for cached questions, and — only when the question bank is empty for a topic — makes a live call to an AI model (Groq or OpenRouter) to generate new ones. Generated questions are stored in MongoDB for future reuse. The system also has a daily quiz cron that regenerates 4 quiz categories at midnight.

02

Issues Found — By Severity

🔴 Cold Start — The #1 Reason Quiz Fails After ~5s CRITICAL
If your backend is on Render's free tier, the server spins down after 15 minutes of inactivity. When a user starts a quiz, the first request hits a sleeping server. Render takes 30–60 seconds to wake it up. Even though your frontend has a 90-second timeout, the request spends most of that budget just waking the server, leaving almost no time for the AI call. This is why it works sometimes (when server is already warm) and fails other times.
server/src/index.ts
🔴 Wrong AI Model — 70B is Too Heavy for Free Tier CRITICAL
Your default model is llama-3.3-70b-versatile. On Groq's free tier (6,000 tokens/minute limit), generating 20 quiz questions with a 70B model consumes ~3,000–5,000 output tokens — nearly your entire per-minute budget in one call. This is why the request often times out. Switch to llama-3.1-8b-instant: it's 10× faster, uses far fewer tokens, and produces nearly identical quiz question quality. 
// ai.service.ts — line ~55
- "llama-3.3-70b-versatile"
+ "llama-3.1-8b-instant"
server/src/services/ai.service.ts
🔴 API Key Not on Deployed Server CRITICAL
You've been switching keys in your local .env file, but this has zero effect on your deployed backend. The server reads environment variables only from the hosting platform's dashboard (Render/Railway Environment tab). Every time you changed providers locally, the deployed server was still using the old key — causing 401 Unauthorized from the AI provider, which your error handler catches and turns into a timeout from the frontend's perspective.
Fix: Go to your hosting dashboard → Environment → add GROQ_API_KEY and AI_PROVIDER=groq explicitly.
🟡 Generating 20 Questions When You Only Need 10 PERFORMANCE
QUIZ_SIZE = 10 in quiz.service.ts, but the AI is called with count: 15 (background backfill) and count: 15 (sync generation). You're asking the AI for 50% more output than you immediately need. This wastes tokens, makes calls slower, and burns through your free-tier quota faster. Change count to 10 to match QUIZ_SIZE.

+ count: 10,  // matches QUIZ_SIZE exactly
server/src/services/quiz.service.ts
🟡 Empty DB Blocks the Entire HTTP Response PERFORMANCE
When a user picks a topic that has never been quizzed before, the code falls into the synchronous generation path — the HTTP response hangs open while the AI generates questions. If the AI call fails (rate limit, timeout, bad key), the user gets an error. The background backfill path (for topics with some questions) is well-designed. The sync path needs a timeout guard and a better UX fallback.
server/src/services/quiz.service.ts · line ~85
🟡 In-Memory Queue Lost on Server Restart RELIABILITY
Your aiQueue.ts stores all jobs in a JavaScript Map in memory. If the server crashes or Render restarts it (which happens frequently on free tier), all queued generation jobs are silently lost. Users who triggered background backfills will get no new questions next time. For now, this is acceptable for a student project, but worth knowing.
server/src/services/aiQueue.ts
🟡 Daily Quiz "Current Affairs" Can't Work With This AI LOGIC BUG
Your CATEGORIES config includes a "Current Affairs" category with prompt: "focus strictly on news from today and yesterday, world events, and politics." But Groq's Llama models have a training cutoff — they have no knowledge of yesterday's news. The AI will generate plausible-sounding but fabricated or stale "current affairs" questions. This is a fundamental mismatch between your feature intent and the AI's capabilities.
Fix: Either remove this category or add a Brave Search / Serper web search step before calling the AI, so it has actual recent news to draw from.
server/src/services/daily-quiz.service.ts · CATEGORIES
🔵 Temperature 0.7 is Too High for Factual Quiz Questions QUALITY
Temperature 0.7 introduces creativity variance — fine for writing, but bad for factual MCQs. It increases the chance the AI invents plausible-but-wrong answer choices or generates questions with debatable correct answers. Lower it to 0.3 for more consistent, accurate quiz content. 
- temperature: 0.7,
+ temperature: 0.3,
🔵 No Retry With Exponential Backoff on Rate Limit (429) MINOR
Your retry logic has a fixed 2-second delay between attempts. If Groq returns a 429 rate limit error, retrying after exactly 2 seconds will likely also fail. Standard practice is exponential backoff: wait 2s, then 4s, then 8s. Also, the error is not inspected — if it's a 429, you know to retry; if it's a 401 (bad key), retrying is pointless.
server/src/services/ai.service.ts · retry loop
✅ What's Already Well-Built GOOD
Zod validation on all endpoints prevents malformed requests. Firebase JWT auth on all protected routes is correct. DB fallback when AI fails is a smart resilience pattern. Rate limiting (aiLimiter: 10 req/min) prevents abuse. Adaptive quiz selection (60% weak concepts, 40% random) is genuinely clever. FIFO cap on attemptedQuestions (MAX_ATTEMPTED=500) prevents unbounded DB growth. Server-side key handling is correct — keys are never exposed to the client.
